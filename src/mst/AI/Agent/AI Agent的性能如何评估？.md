首先我希望先明确一个定义，那就是 AI Agent 的性能评估不仅仅是为了证明你的 Agent 有多好，更是为了**方便你找到改进的方向**。
AI Agent 的性能评估是多方面多维度的，比如会从下面的这些维度来进行评估：**任务完成率**、**准确度**、**响应时间**、**用户评价与转化率**。

这几个评价维度很好理解，我就不去一一解释什么意思了，我重点说一下**准确度**，准确度其实是分为两个维度的，第一个维度是对用户问题的语义理解准确度，第二个维度是生成结果的准确度。

AI Agent 的评估往往是和你的 Agent 的应用方向有密切的关系，比如在 **QA 问答类型 Agent 上，可能我们更加看中的是响应时间**。在**推荐类型 Agent 上我们可能更加看重的是推荐的准确率和转化率**。在**客服类型 Agent 上，我们更加看重的是用户的评价和任务完成率**。

---

#### AI Agent 评估系统的建设

要格外注意，AI Agent 的评估不是一次性的，而是要从 Agent 开发初始就一直进行评估，我们要建立一套**实时**、**高效**、**可配置的** Agent 评估监控系统。
能够对各个阶段的 Agent 进行系统评估，比如在**开发阶段更看重准确性**，**测试阶段看中稳定性和性能**，在**生产阶段看中用户的满意度和转化率**。

具体到评估系统设计上我们就要考虑这么几个问题：
① <font color=red>**评估监控不能够影响业务的执行和性能**</font>

② **评价监控系统要能够应对数据量的爆炸式增长**

③ 评价监控系统能够**对比不同版本的 Agent 的指标变化**

④ **要有很好的扩展性**

这四点是一个 Agent 评价监控系统的基本要求，下面针对这四个点来分开说明

1. **评价监控不能影响业务的执行和性能**

​ 我们的这个系统不能对原始业务产生影响，比如我们的这个监控系统出了问题导致整个原始业务宕机了，而且也不能对原始业务的性能有太大的影响。所以我们可以考虑使用**消息队列+异步发送**的方案来收集 Agent 执行全过程的日志，比如执行开始、执行结束这个过程中的**日志**。监控系统只需要分析这些日志就可以了。

2. **评价监控系统要能够应对数据量的爆炸式增长**

​ 随着我们公司中的 Agent 越来越多，对应的日志就会越来越多，我们要能够应对这种爆炸式的日志增长。怎么解决呢：我们可以进行冷热数据分离，比如我们将最近一个季度的日志放在高性能库中，之前的数据放在性能差点的库中。

3. **评价监控系统能够对比不同版本的 Agent 的指标变化**

​ 我们的这个系统要能够做到在我们优化 Agent 之后能够对比不同版本的 Agent 的性能变化，这个需求可能还需要借助路由来做，比如我们可以根据用户的 userId 来做一个 Hash 算法，让同一个用户的请求会到固定的一组机器，然后这组机器采用 A/B 测试，来看一下不同版本的 Agent 的性能变化。

1. **要有很好的扩展性**

​ 评价系统要有随业务变化的能力，随着发展，我们的 Agent 可能由纯本文 Agent 变成了多模态 Agent，那么你此时可能就能够从文本、图片、音频多个维度来监控准确性了。所以在设计之初就做好扩展性，减少硬编码。

::: tip

**公司中一定要做好测试集**。因为**用户的问题类型可能会突然发生变化**、**模型也可能会出现退化问题**。在出现问题之后我们优化之后需要利用测试集快速回归测试。

:::

---

写在最后：

我们在进行 Agent 性能评估的时候要注意，我们的 Agent 的性能评估**不要脱离业务**，不同的业务看中的指标是不一样的，不要想着一下子做出来一个十全十美的“完美 Agent",要记住：“**Agent 的优化过程是 ing 进行时**”。而且我们在进行 Agent 性能优化的时候**要考虑到优化精度和成本的均衡**，比如我们花了很大的精力将响应时间从 200ms 优化到了 190ms，从数值上看确实性能有所提高，但是对于业务来说，用户感觉不到明显变化，并没有带来什么效益反而增加了开发成本。
