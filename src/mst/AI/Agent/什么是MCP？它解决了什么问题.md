## 什么是 MCP？它解决了什么问题

MCP 又叫做模型上下文协议，本身是一种协议。本质是一个连接层的协议。主要目的是**让 AI 模型与外部数据源、工具以一种统一的数据格式来进行交互**，**解决 AI 大模型与外部世界交互的标准化缺失**。

比如在实际的公司业务中的体现：当用户在我们的 AI 聊天口询问订单信息的时候，AI 模型可以去查询商品系统、订单系统、物流系统、客服模块的数据，然后将结果告知用户。

在 MCP 出现之前，我们只能够使用硬编码的方式来让 AI 模型调用我们的各个系统的查询，那么我们需要在每一个系统中都要针对不同的 AI 模型去写一套**连接器以及安全校验机制**。有了 MCP 之后，我们只需要让我们的系统按照 MCP 协议的标准，来**将我们的服务变成 MCP Server**,那么我们就能支持所有的 AI 模型通过 MCP 来调用我们系统。

MCP 其实是分为三层架构来设计的，分别是：客户端层、服务端层、传输层。
客户端层： 负责对 AI 模型的请求进行封装以及响应的解析
服务端层： 负责逻辑的实现
传输层：负责协议通信和安全认证

---

#### MCP 和 API 有什么区别？

API 只负责系统间的数据传输。而 MCP 除了实现系统间的数据传输之外，还有一个特点，那就是：带有<font color=red>**语义理解的上下文传递**</font>。比如我们在查询出商品的信息来之后，MCP 协议会进行语义理解，它会理解数据对应的业务，比如什么是促销价格、什么是库存状态等。

---

#### MCP 安全机制

如果我们使用 MCP，有一个我们非常关心的指标，那就是安全性。大家可以考虑一下，如果我设计了一个 MCP Server,支持 MCP Client 来访问我的服务，如果我没有做好安全认证机制，那么任何人都有权限来看我的系统中的数据，甚至可以通过 MCP Client 来对我的数据进行删除和修改。那么对于一个企业级项目来说，这是不可能会接受的。
而 MCP 有多层安全机制

1. 身份认证
2. 权限控制
3. 数据访问审计
4. 操作边界限制

::: tip info
其中，数据访问审计和操作边界限制是可以通过沙箱（SandBox）实现的。
:::

Spring AI Alibaba 目前就可以通过 OAuth2 和 Api 密钥的方式来进行安全认证。

我下面将用一个现实世界中的小 case，帮助你更好的理解 MCP 的安全机制
一个公司中有这么几类人，boss(AI 模型)、外包人员/工具（MCP Server）、行政、审计（Host、Client）
当一个 Boss 说，使用高德地图（MCP Server）帮我规划一个旅游计划，**它只负责"说",但是它没有权利直接去操作高德地图**。
Boss 发命令之后,行政、审计人员去审核和操作外包人员（Mcp Server），因为这个外包人员可能会说，我还会读文件，此时审计（Host、Client）可能会说“我让你读了吗？”，最终决定权在审计（Host、Client）
还有一种情况，就是一些恶意的 MCP Server 会去返回一些恶意内容，骗 AI 去干坏事，此时它只能让 AI 去说，但是实际干不干可不是由 AI 决定的，而是有 Host、Client 决定。而 Host、Client 就是一种沙箱，来限制 AI 操作的边界。

总结一下：
BOSS: 帮我规划一下从北京到上海的路线
1️⃣ AI：

> “我想用 route_planner 这个工具”

2️⃣ Host：

> “行，我看看你有没有权限”

3️⃣ Host 检查：

> 工具是不是白名单？

> 只能访问高德 API 吗？

> 参数有没有越界？

4️⃣ Host 替你去调用 高德 MCP Server

5️⃣ 高德返回结果

6️⃣ Host 再把 “过滤后的结果” 给 AI

👉 **AI 从头到尾没碰过网络**
